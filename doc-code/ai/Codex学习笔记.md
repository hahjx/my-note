# Codex 学习笔记

## 1. 三层配置怎么理解

可以先用这个模型记：

```text
config.toml = 能不能做、允许做到什么程度
AGENTS.md   = 应该怎么做、按什么流程做
Prompt      = 这一次具体要做什么
```

`config.toml` 更像车辆的硬性配置，负责模型、权限、沙箱、联网、MCP、Hooks 等能力边界。

`AGENTS.md` 更像给驾驶员的工作规程，负责当前项目或目录中的协作习惯、开发流程、校验方式和输出格式。

对话里的提示词是本次任务的临时要求，优先级很高，但只影响当前对话或当前任务。

## 2. 前端开发时建议怎么用

前端项目里，建议优先把稳定规则写进 `AGENTS.md`：

- 页面定位优先搜索可见文案、路由、按钮文案、表格列名和弹窗标题。
- 修改代码前先阅读相关页面、组件和服务请求。
- 复杂需求先输出实现和校验步骤，等确认后再改代码。
- 改已有页面时沿用现有组件库、接口风格和目录结构。
- 完成后说明改了什么、没改什么、如何验证、还有什么风险。

`config.toml` 不适合写业务流程，它适合控制权限。例如是否允许联网、是否允许写工作区外文件、是否需要执行命令前确认。

## 3. Skill 是什么

Skill 可以理解成 Codex 的可复用工作流包。它通常是一个目录，核心文件是 `SKILL.md`，可以额外带模板、脚本、参考资料。

适合做成 Skill 的内容，一般有这些特点：

- 跨多个项目都会用到。
- 步骤比较固定。
- 需要一套专门的方法论或检查清单。
- 可能会配套脚本、模板或参考文档。

前端开发可以考虑沉淀这些 Skill：

- `frontend-page-change`：页面需求定位、影响分析、实现、校验。
- `ui-check`：启动页面后截图检查布局、交互、响应式。
- `release-checklist`：提交前检查 lint、build、test、分支、PR 和部署说明。
- `codex-note`：把和 Codex 学到的内容整理成结构化笔记。

## 4. 工作区和附件的区别

Codex App 左下角的 `+` 更像是把某个文件添加到当前对话上下文，适合临时让 Codex 看一个文件。

工作区则是 Codex 真正执行任务的位置，会影响：

- 终端默认目录。
- 能扫描和修改的项目文件。
- 是否读取当前目录下的 `AGENTS.md`。
- git 状态、分支、提交和测试命令。
- 项目级 `.codex/config.toml` 是否生效。

所以文件夹通常需要通过 Codex App 的项目或工作区选择入口添加，而不是通过附件按钮添加。

## 5. 建议的学习方式

每学到一个 Codex 概念，可以按这个格式记录：

```text
概念：
我现在的理解：
适合放在哪里：
什么时候用：
不要怎么用：
例子：
```

这样后续可以把零散理解慢慢整理成自己的 Codex 使用手册。
